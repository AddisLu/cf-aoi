目的：相機與交換器尚未到貨前，先用舊機台存圖把「RDMA 傳輸 ＋ Spark AOI 運算」整條流程跑起來，確認會不會塞車、一台 Spark 夠不夠。

一句話結論
  • 不塞車 全量 744 張（24 CCD × 31）跑完，佇列全程是空的、零背壓、零錯誤。瓶頸在「傳輸」不在「運算」。
  • 一台 Spark 夠 以實測 7.4ms/張推算，一面板（37 CCD × ~30 張 ≈ 1110 張）GPU 運算約 8.2 秒，遠低於 20–30s 節拍；缺陷 <100 顆屬輕負載，更游刃有餘。（前提見下方產能分析）
  • 配方待接手調 目前的 baseline 配方「pitch 沒對準面板週期」，把整片像素陣列都當成缺陷（多數張數頂到 10000 上限），所以這一輪「真實缺陷數」還不能用，也分不出你說有缺陷的 IP04 第 27、28 張。需要拿舊機台已調好的配方來移植。

解讀：GPU 一張只要 ~7–8ms，但每張 40.8MB 在目前 MTU 1500 下要 ~0.48 秒才傳完，所以整條鏈路是「傳輸受限」——Spark 收到圖幾乎立刻就處理完、接著閒著等下一張。因此 FrameQueue 全程是空的（峰值 0/8）、完全沒有背壓、744 張零錯誤。運算端遠遠沒被塞住。

(b) RDMA 鏈路已持久化

放假期間網路線被改插過（原本的 enp1s0f0np0 已無載波），這也是先前 RDMA 連線被 REJECT 的真正原因——不是程式、也不是版本問題（已用程式碼 diff ＋ 版本比對排除）。本次把 192.168.3.1 重新綁到實際接線的 enp1s0f1np1，並用 NetworkManager 建了固定設定檔 cf-rdma（開機自動、載波恢復會自動重套），整段 744 張跑完中途零斷線。註：若日後又換插槽，需把設定檔的介面名一起改。

你指出的 IP04 第 27、28 張：本輪測得 617 / 8155 顆——但同一顆 CCD 其它「乾淨」張數也都好幾千，所以分不出來。

原因：DIV 演算法是拿每個點跟「相隔一個 pitch 的同相位鄰點」比值；pitch 一旦沒對準面板的像素週期，整片規則陣列都會被誤判成缺陷。33/17 不是 T550QVN10 的正確週期。我也試過用 FFT 盲抓 pitch，但每顆 CCD 看的是面板不同區域、取樣處的週期性不明顯，沒辦法盲調 24 顆。

正確做法：移植，不要重猜。 本專案策略一向是「從舊系統移植、不重寫」。舊機台本來就有一份調好的 T550QVN10 配方（正確的 pitch / ROI / 閾值 / 對位，當年就是它抓到 IP04 27、28 的缺陷）。
  • 請提供舊機台的 RecipeInfo.xml（各 CCD/IP 的）→ 我直接移植進 recipes/T550QVN10_TGT_G/IPxx/（同一套 legacy Recipe 格式），重跑後 IP04 27、28 就能跟乾淨張數明顯分開。
  • 若手邊沒有，改走 Control Step1 逐 CCD 人工調（載入影像 → 用 FFT 量 pitch → 框出陣列 ROI → 設對位 Mark），這本來就是設計上的 per-CCD 調校流程，我可以陪你一步步做。

二之重點 ── 產能分析：一台 Spark 夠不夠？

夠。一台 Spark 在「運算」上充裕。

缺陷 <100 顆是輕負載情境（缺陷少 → blob 分析與存圖負擔小），GPU 每張落在 ~7ms 低端。以一面板的張數推算：

情境張數/面板GPU/張單台序列總時間佔 20–30s 節拍結論
生產（37 CCD × ~30 張）＝專案投影~11107.4 ms~8.2 s27–41%OK，餘裕 ~60–73%
本次舊資料（24 CCD × 31 張）7447.4 ms~5.5 s18–28%OK

G8 與 G8.5 的差別主要在「掃描張數」（玻璃尺寸 → slice 數略不同），量級都在每顆 CCD ~30 張，總時間同在 ~8 秒上下，兩者都遠低於 20–30s 節拍。確切張數需以實際掃描幾何為準，但即使翻倍（~16s）仍在節拍內。

數字依據：7.4ms/張為實測，本次 744 張真實圖再次確認 GPU 落在 7–8ms（即便 baseline 配方爆出上千顆缺陷、GPU 也只 ~8ms）；缺陷 <100 顆會更快。整段佇列空、零背壓 → 運算端不是瓶頸。

⚠ 但有一個前提：傳輸要用 jumbo MTU + 真實陣列並行

  • 本次回放量到的 2.1 fps（85.8 MB/s）是「回放工具在 MTU 1500、單一資料流」的限制，不是生產傳輸的真實速度。若照這個數字，1110 張要 ~9 分鐘——那會超節拍。
  • 生產時是 37 顆 CCD 經交換器「同時」串流、且鏈路該開 jumbo MTU 9000（Phase-1 已實測 100G 線速、延遲 1.44µs）。在這條件下傳輸吞吐遠高於 GPU 速度，瓶頸回到 GPU（~8.2s/面板），一台 Spark 即足。
  • 本次因 damac 端改 MTU 需要 sudo 密碼（我這邊拿不到），維持 1500 對稱跑通，故吞吐偏低——這只影響「這次回放」，不影響上面的產能結論。建議正式跑貨前把兩端都設成 MTU 9000。

項目狀態後續

(c) 真實缺陷偵測（含 IP04 27/28）待配方提供舊機台 T550QVN10 配方讓我移植；或在 Control Step1 逐 CCD 調
