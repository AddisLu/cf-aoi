RDMA 回放驗證報告

機種 T550QVN10_TGT_G　·　2026-06-23　·　截取中心(damac) → RDMA → Spark(IP) 運算
目的：相機與交換器尚未到貨前，先用舊機台存圖把「RDMA 傳輸 ＋ Spark AOI 運算」整條流程跑起來，確認會不會塞車、一台 Spark 夠不夠。

一句話結論
  • 不塞車 全量 744 張（24 CCD × 31）跑完，收端佇列全程是空的、零背壓、零錯誤；GPU 不是瓶頸。
  • 找到並修掉真瓶頸（15.7×） 原本傳輸卡在 86MB/s——根因是 crc32_ieee 逐位元（非 MTU、非 fabric：MTU 9000 無改善、原生 RDMA 達 11.4GB/s）。改 slice-by-8 後 85.8→1346 MB/s、2.2→34.6 fps（commit b9d7d16）。
  • 24CCD 一台 Spark 夠 / 37CCD 臨界 修正後管線 34.6fps：744 張 ~21.5s（節拍內）；1110 張 ~32s（略超 30s，GPU 6.8ms 仍有大餘裕，收端管線再優化即穩進）。缺陷 <100 顆為輕負載。
  • 配方待接手調 目前的 baseline 配方「pitch 沒對準面板週期」，把整片像素陣列都當成缺陷（多數張數頂到 10000 上限），所以這一輪「真實缺陷數」還不能用，也分不出你說有缺陷的 IP04 第 27、28 張。需要拿舊機台已調好的配方來移植。

(b) RDMA 鏈路已持久化

放假期間網路線被改插過（原本的 enp1s0f0np0 已無載波），這也是先前 RDMA 連線被 REJECT 的真正原因——不是程式、也不是版本問題（已用程式碼 diff ＋ 版本比對排除）。本次把 192.168.3.1 重新綁到實際接線的 enp1s0f1np1，並用 NetworkManager 建了固定設定檔 cf-rdma（開機自動、載波恢復會自動重套），整段 744 張跑完中途零斷線。註：若日後又換插槽，需把設定檔的介面名一起改。

你指出的 IP04 第 27、28 張：本輪測得 617 / 8155 顆——但同一顆 CCD 其它「乾淨」張數也都好幾千，所以分不出來。

原因：DIV 演算法是拿每個點跟「相隔一個 pitch 的同相位鄰點」比值；pitch 一旦沒對準面板的像素週期，整片規則陣列都會被誤判成缺陷。33/17 不是 T550QVN10 的正確週期。我也試過用 FFT 盲抓 pitch，但每顆 CCD 看的是面板不同區域、取樣處的週期性不明顯，沒辦法盲調 24 顆。

正確做法：移植，不要重猜。 本專案策略一向是「從舊系統移植、不重寫」。舊機台本來就有一份調好的 T550QVN10 配方（正確的 pitch / ROI / 閾值 / 對位，當年就是它抓到 IP04 27、28 的缺陷）。
  • 請提供舊機台的 RecipeInfo.xml（各 CCD/IP 的）→ 我直接移植進 recipes/T550QVN10_TGT_G/IPxx/（同一套 legacy Recipe 格式），重跑後 IP04 27、28 就能跟乾淨張數明顯分開。
  • 若手邊沒有，改走 Control Step1 逐 CCD 人工調（載入影像 → 用 FFT 量 pitch → 框出陣列 ROI → 設對位 Mark），這本來就是設計上的 per-CCD 調校流程，我可以陪你一步步做。

二之重點 ── 產能分析：一台 Spark 夠不夠？

夠。一台 Spark 在「運算」上充裕。

缺陷 <100 顆是輕負載情境（缺陷少 → blob 分析與存圖負擔小），GPU 每張落在 ~7ms 低端。以一面板的張數推算：

情境張數/面板GPU/張單台序列總時間佔 20–30s 節拍結論
生產（37 CCD × ~30 張）＝專案投影~11107.4 ms~8.2 s27–41%OK，餘裕 ~60–73%
本次舊資料（24 CCD × 31 張）7447.4 ms~5.5 s18–28%OK

G8 與 G8.5 的差別主要在「掃描張數」（玻璃尺寸 → slice 數略不同），量級都在每顆 CCD ~30 張，總時間同在 ~8 秒上下，兩者都遠低於 20–30s 節拍。確切張數需以實際掃描幾何為準，但即使翻倍（~16s）仍在節拍內。

數字依據：7.4ms/張為實測，本次 744 張真實圖再次確認 GPU 落在 7–8ms（即便 baseline 配方爆出上千顆缺陷、GPU 也只 ~8ms）；缺陷 <100 顆會更快。整段佇列空、零背壓 → 運算端不是瓶頸。

傳輸面：原本卡在 86MB/s，根因不是 MTU 而是 CRC（已修，15.7×）

第一版報告把傳輸瓶頸推給 MTU；實測後證明錯了，在此更正並附根因：

步驟吞吐結論
MTU 1500 全幅回放85.8 MB/s（2.2 fps）基準
MTU 9000 兩端 + 8972B DF ping 通85.8 MB/s（不變）→ MTU 不是瓶頸
ib_write_bw 原生 RDMA11,682 MB/s（~93 Gbps）→ 100G fabric 完全健康
換 ring 記憶體（cudaHostAlloc→一般）85.9 MB/s（不變）→ 記憶體也不是
根因：crc32_ieee 為逐位元（8 iter/byte ≈ 86MB/s），每幀 40.8MB 收/送兩端各算一次≈ 86 MB/s= 天花板（256×256 的 1375fps×65KB 也是 86MB/s，吻合）
修正：改 slice-by-8 表驅動（同 IEEE、wire 相容、驗 CBF43926 PASS）1346 MB/s（34.6 fps）15.7× ↑ commit b9d7d16

說明：CRC 修好後 RDMA 收圖管線（單一資料流、真實 frames-via-RDMA）達 34.6fps / 1.35GB/s，佇列仍 0/8、零背壓。 全量 744 張用「回放工具」量到 2.93fps，是 Python(PIL) 逐張解 TIF + pipe 的工具限制（生產相機是直接串 raw、無此解碼），非管線真實速度，故產能以 34.6fps 管線值估算。

修正後的產能結論

情境張數/面板@34.6fps 管線佔 20–30s 節拍結論
本次舊資料（24 CCD × 31）744~21.5 s72–100%節拍內
生產（37 CCD × ~30）~1110~32 s略超 30s臨界
  • GPU 不是瓶頸：每張 6.8–8.1ms（124–147fps 能力）。目前 34.6fps 的限制是收端「CRC 驗證 + 40.8MB memcpy」與同步 N-slot 協議的串接負擔，還可再優化（slice-by-16 / 硬體 CRC、或省去收端冗餘 CRC〔RDMA RC 本就保證送達〕、或非同步多筆 in-flight）→ 趨近 GPU 上限 147fps → 1110 張 ~7.5s，餘裕充足。
  • 缺陷 <100 顆 = 輕負載（GPU ~7ms 低端），上表已是此情境。
  • 生產為 37 CCD 經交換器並行餵入，但收端是單台 Spark 單一處理序，故吞吐看收端序列速率（34.6fps，優化後更高）。

一台 Spark：24CCD/744 張在節拍內 ✅；37CCD/1110 張臨界（目前 ~32s），GPU 有大量餘裕、收端管線再優化即穩進節拍。
