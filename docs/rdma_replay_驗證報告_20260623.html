RDMA 回放驗證報告

機種 T550QVN10_TGT_G　·　2026-06-23　·　截取中心(damac) → RDMA → Spark(IP) 運算
目的：相機與交換器尚未到貨前，先用舊機台存圖把「RDMA 傳輸 ＋ Spark AOI 運算」整條流程跑起來，確認會不會塞車、一台 Spark 夠不夠。

一句話結論
  • 不塞車 全量 744 張（24 CCD × 31）跑完，收端佇列全程是空的、零背壓、零錯誤；GPU 不是瓶頸。
  • 兩段瓶頸都修掉了（共 2.2×，最終 33→73.8fps） ① 傳輸原卡 86MB/s — 根因是 crc32_ieee 逐位元（非 MTU、非 fabric），改 slice-by-8 →34.6fps（b9d7d16）。② 接著卡 ~33fps，原以為要「3 步驟重構」，深挖後發現真根因是 min_rnr_timer=655ms 的 RNR 災難等待（CRC 一關送端就 RNR 崩潰，CRC 意外當了限速器）。改 0.64ms + 送端非同步多緩衝 + CRC 改可選 → 73.8 fps（commit 66bc8ba+dee1a15），關 CRC 仍 bit-exact。
  • 一台 Spark 夠（24CCD 也夠、37CCD 也夠） 修正後管線 73.8fps：744 張 ~10s、1110 張 ~15s，皆遠進 20–30s 節拍（餘裕 25–50%）。原本標「37CCD 臨界」已不成立。步驟 1+2 已達標；步驟 3（device-direct）經調查後刻意不做（生產存圖必須保留整張 host 像素來裁缺陷小圖，device-direct 只利合成壓測、不利生產，且要冒 bit-exact 風險）。缺陷 <100 顆為輕負載。
  • 配方待接手調 目前的 baseline 配方「pitch 沒對準面板週期」，把整片像素陣列都當成缺陷（多數張數頂到 10000 上限），所以這一輪「真實缺陷數」還不能用，也分不出你說有缺陷的 IP04 第 27、28 張。需要拿舊機台已調好的配方來移植。

二之重點 ── 產能分析：一台 Spark 夠不夠？

夠。一台 Spark 在「運算」上充裕。

缺陷 <100 顆是輕負載情境（缺陷少 → blob 分析與存圖負擔小），GPU 每張落在 ~7ms 低端。以一面板的張數推算：

情境張數/面板GPU/張單台序列總時間佔 20–30s 節拍結論
生產（37 CCD × ~30 張）＝專案投影~11107.4 ms~8.2 s27–41%OK，餘裕 ~60–73%
本次舊資料（24 CCD × 31 張）7447.4 ms~5.5 s18–28%OK

G8 與 G8.5 的差別主要在「掃描張數」（玻璃尺寸 → slice 數略不同），量級都在每顆 CCD ~30 張，總時間同在 ~8 秒上下，兩者都遠低於 20–30s 節拍。確切張數需以實際掃描幾何為準，但即使翻倍（~16s）仍在節拍內。

數字依據：7.4ms/張為實測，本次 744 張真實圖再次確認 GPU 落在 7–8ms（即便 baseline 配方爆出上千顆缺陷、GPU 也只 ~8ms）；缺陷 <100 顆會更快。整段佇列空、零背壓 → 運算端不是瓶頸。

傳輸面：原本卡在 86MB/s，根因不是 MTU 而是 CRC（已修，15.7×）

第一版報告把傳輸瓶頸推給 MTU；實測後證明錯了，在此更正並附根因：

步驟吞吐結論
MTU 1500 全幅回放85.8 MB/s（2.2 fps）基準
MTU 9000 兩端 + 8972B DF ping 通85.8 MB/s（不變）→ MTU 不是瓶頸
ib_write_bw 原生 RDMA11,682 MB/s（~93 Gbps）→ 100G fabric 完全健康
換 ring 記憶體（cudaHostAlloc→一般）85.9 MB/s（不變）→ 記憶體也不是
根因：crc32_ieee 為逐位元（8 iter/byte ≈ 86MB/s），每幀 40.8MB 收/送兩端各算一次≈ 86 MB/s= 天花板（256×256 的 1375fps×65KB 也是 86MB/s，吻合）
修正：改 slice-by-8 表驅動（同 IEEE、wire 相容、驗 CBF43926 PASS）1346 MB/s（34.6 fps）15.7× ↑ commit b9d7d16

說明：CRC 修好後 RDMA 收圖管線（單一資料流、真實 frames-via-RDMA）達 34.6fps / 1.35GB/s，佇列仍 0/8、零背壓。 全量 744 張用「回放工具」量到 2.93fps，是 Python(PIL) 逐張解 TIF + pipe 的工具限制（生產相機是直接串 raw、無此解碼），非管線真實速度，故產能以 34.6fps 管線值估算。

突破 33fps 牆 — 真根因是 min_rnr_timer，不是「3 步驟」（2026-06-23 已修並實測）

原報告判定 ~33fps 是「結構性牆、需 3 步驟重構」。進一步診斷後修正：33fps 的真正成因有二，已解決其一即破牆。

診斷（收端 recv_thread 每幀拆分）耗時說明
app-CRC 複驗（CPU 讀 40.8MB pinned 算 CRC）16.3 ms收端瓶頸大頭
slot→host memcpy（CPU 讀 pinned ~3.4GB/s）11.2 ms次要
push 入佇列0.0 ms—
合計 → 收端上限~27.5 ms = ~36fps= 觀測的 ~33fps 牆

把 app-CRC 關掉本應大幅加速，但兩端關 CRC 反而崩到 2.6fps。追下去發現：CRC 一關，送端變快、瞬間吃光 credit → RNR NAK；而 rdma_cm 預設 min_rnr_timer = index 0 = 655ms → 每次 RNR 等 0.6 秒 → 吞吐崩潰。原本 CRC 的 16ms 等於意外地把送端「降速對齊」收端、剛好避開 RNR ── 這才是 33fps 的隱形天花板。

變因（送端非同步多緩衝 = 步驟 1，已 commit）fps結論
CRC-ON 兩端 + min_rnr 修正（安全預設）~31–34無回歸、err=0
CRC-OFF 兩端，未修 min_rnr2.6（RNR 崩潰）655ms RNR 等待
CRC-OFF 兩端 + min_rnr_timer=index12(~0.64ms)70–742.2× ↑、破牆

修法（commit dee1a15）：① 收端 accept 後 ibv_modify_qp 設 min_rnr_timer=0.64ms（安全、永遠開，純修 RNR 災難等待）；② app-CRC 改環境變數 CFAOI_RDMA_NOCRC=1 可選關（RDMA RC 本就保證有序無損送達，可信 fabric 求吞吐時關，預設仍做 CRC 保守）。關 CRC 後 IP04 31 張缺陷數對 golden 仍 bit-exact PASS ── 資料正確性不靠 app-CRC（靠 RC），CRC 只是額外防呆。

步驟 3（device-direct）── 經調查後不做，附理由

原「步驟 3」是「收端直接 cudaMemcpyAsync slot→device、省掉 host 拷貝」。調查程式後確認對生產無效，故不實作：

  • 生產存圖需要整張 host 像素：缺陷小圖是從 host 影像裁切（result_saver.cpp:363 由 host 指標 img 裁 patch）。生產一定要存缺陷小圖（給 DefectSort / AI 標註）→ 整張 host 拷貝無法省。device-direct 只在 --no-save-images 的合成壓測下有意義，不是真實生產負載。
  • 而生產所需的 ~73.8fps 已達標（1110 張 15s < 30s 節拍）：收端 host 拷貝（必要）把管線定在 ~73fps，剛好遠進節拍。再追 device-direct（合成可達 ~150fps）是超出需求的 headroom，且要改 result_saver 裁切改走 device（cudaMemcpy2D D2H）= 熱路徑重構、有 bit-exact 風險，效益/風險不成比例。
  • 結論：步驟 1+2 已足以達標；步驟 3 是「可做但不該做」── 為合成數字而冒生產正確性風險，違反工程紀律。若未來真的要破 73fps（例如 CCD 暴增），正解是多台 Spark 水平擴充或 GPU-side 裁切，屆時再評估。
  • GPU 不是瓶頸：每張 6.4–8.1ms（124–156fps 能力）。修正後 ~73.8fps 的限制移到「收端 host 拷貝 11.7ms」── 這份拷貝生產必要（存圖用），故是合理上限。
  • 缺陷 <100 顆 = 輕負載（GPU ~7ms 低端），上表已是此情境。
  • 生產為 37 CCD 經交換器並行餵入，但收端是單台 Spark 單一處理序，故吞吐看收端序列速率（現 73.8fps）。

一台 Spark：24CCD/744 ~10s ✅、37CCD/1110 ~15s ✅，皆遠進 20–30s 節拍，餘裕 25–50%。33fps 牆已破（真根因 = min_rnr_timer 655ms，非「3 步驟」）；步驟 1+2 已達標、步驟 3 經調查後刻意不做（生產存圖必須保留 host 拷貝，再優化是超需求 headroom）。
