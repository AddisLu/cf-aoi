一、全 GPU 可行性

偵測核心 100% 在 GPU：DIV(kernelFast8WayComparison)/SUB(kernelSub8WayStarVoting)/融合(kernelDivVoting8WayStar) → CCL(kernelFastCCL*) → blob(kernelFusedFlattenBlobStats)；前處理 LSC/remap/高斯平滑/多尺度 downsample-upscale 皆 kernel。CPU 殘留：remap 256-bin 直方圖 min/max readback(每幀一次 sync，可改 GPU reduction)、CCL 收斂判斷(改 device-loop 風險高、觸 bit-exact 不變式，建議不動)、recipe 解析/結果下載/defect_rules 後處理(幀邊界、量小，留 CPU 合理)。結論：演算法本體可全 GPU，且現已是。

三、融合演算法（kernelDivVoting8WayStar, mode2）

保留 legacy 投票骨架(16 路=8方向×PitchTimes、每路 3×3 SAD 局部匹配、ChooseAmount 多數決)，但每路比較改用 DIV 比值，整數定點 center·1024 vs neighbor·round(TH·1024) 避除法。融合各家優點：

融合點來源作用
每路 DIV 比值新 Demo背光/照度乘性不變
16 路 ChooseAmount 投票 + 3×3 SAD舊 SUBpattern/雜訊/skew robustness
LSC 徑向校正(前處理,可選)新 Demo鏡頭 vignetting
暗區棄權(鄰點<dark_eps 該路不投)新增(對抗複查定案)邊緣 FP 改善①：杜絕暗邊界比值爆衝
比例投票門檻(need=⌈choose/16×valid⌉)新增邊緣 FP 改善②：邊緣抽路時靈敏度一致(不漏不誤)
有效路<過半→判背景新增邊緣 FP 改善③：角落/深暗兜底
多尺度 resize-redetect(2×/4×)新 Demo(原休眠,本次接線)大顆 Defect 補強

⚠️ remap 只用於 SUB(mode1)：它減 min 會破壞比值 center/neighbor 的照度不變性，故 DIV-voting 不做 remap。

五、產線建議與待收尾

  • 跑片用融合 mode2：recipe 以 M_AlgorithmWayCompare=Awc_8_Way_Star_Div 標記 → 守門自動走 mode2；閾值直接給比值域(BrightThreshold>1、DarkThreshold<1，如 1.2/0.8)，不經 SUB→R 換算。
  • Step E Blob 過濾(BlobMinSize/MergeDistance)：壓 size<3 雜訊 FP(實測 338→65)、合併鄰近 → 必做。
  • 4× 多尺度除錯：multiscale=2 結果異常(maxSize=2)，4× downsample/upscale 或 blob-on-large-region 疑 bug → 暫用 2×。
  • 閾值 per-recipe 調：DTH=0.80 保 027+028;0.75 漏 028(弱)。產線依缺陷規格 + (若有)舊機台 golden 逐 CCD 調。
  • LSC 標定：開 LSC 需先標 k1/k2/k3(多環擬合)+ 跨架構一致性驗證(改像素值會動邊界，須重驗不變式 12)。
  • remap min/max GPU 化(可選優化)：消每幀一次 stream sync。

融合演算法已可跑片(mode2)，集 DIV 照度不變 + SUB 投票 robustness + LSC + 邊緣 FP 改善 + 大顆補強於一身、全 GPU；產線前補 Blob 過濾即收斂 FP。
