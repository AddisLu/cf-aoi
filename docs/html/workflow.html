工作流程 — 人 × Claude 協作開發

這套系統不是從空白重寫,而是從一份舊版 WinForm 單體軟體, 在人與 Claude 的分工協作下,經「底層測試先行 → 拆解三模塊 → 考古/實作/驗證 → 由離線讀圖逐步接上相機取像中心」 一路演進而成。本頁說明這個流程怎麼跑。

方法論文件 · 對應 docs/CLAUDE.md §3/§4/§8 + STATUS.md L0–L4 分級

方法論一覽

核心三原則:

遷移,不重寫

從 Reference/(舊 WinForm + Demo GPU + phase1 測試)遷移既有程式碼,GPU kernel 本體一字不改,只換 I/O 外殼。

底層先驗

先寫單體測試把「取像核心」「GPU 核心」「RDMA 鏈路」各自驗到綠,再往上組裝;不在未驗的地基上蓋樓。

誠實分級

每個功能標 L0–L4,「寫好 ≠ 驗證過」。沒貼出實機數據前不算完成。

人 × Claude 分工

兩邊各做自己擅長的事,以「實機數據 + 誠實分級」為交接介面。

整體開發流程

從舊單體到生產線,大方向是一條由下往上、由離線往實機的演進路徑:

phase1 套件把「能取像」「能算對」「能傳對」各自驗到 L4,再往上組

① 底層單體測試先行(取像核心 + GPU 核心)

動架構之前,先確定三塊地基各自可信。每塊用獨立的小測試程式驗到實機 L4,排除「上層出錯到底是誰的鍋」。

地基單體測試驗什麼實機結果
取像核心t30_pylon_probe / t31_pylon_grab相機偵測 + 500 幀零掉幀 + 真感光L4 0.0000% drop
RDMA 鏈路10_rdma_linkcheck / ib_write_lat100G PORT_ACTIVE、RoCE v2、延遲L4 avg 1.44μs
取像→GPUt21_rdma_gpu / t40_e2e32MB 逐位元 CRC 兩端一致L4 CRC 零錯
GPU 核心--verify-deterministic同影像兩跑 bit-exactL4 x86 & GB10 全 bit-exact
為何先做這個取像核心(相機真的有感光、不掉幀)與 GPU 核心(演算法真的算對、可重現)是整個系統的兩根支柱。 支柱用 phase1 小程式單獨驗綠後,後面拆模塊、接線出問題時,可以排除「地基壞掉」這個可能。

② 分析原始軟體 → 拆解為 3 模塊

讀懂 legacy 單體在做的事,沿「機器邊界」切成三支可獨立部署的程式;MIL/相機/GPU 不再綁在一起。

拆解後,跨機介面變成明確協議:上位機 CF_(8787)、Control↔Grab/IP JSON(8100/8200)、Grab↔IP RDMA。 MIL/FrameGrabber/CamProc 完全移除,改 pylon + RDMA + Avalonia。

③ 每模塊:考古(plan)→ 實作(coding)→ 驗證(testing)

三模塊各自跑同一個三段循環。這也是每次「新功能 / 補 gap」的標準節奏。

考古 Plan

  • 逐檔讀 Reference/(WinForm / Demo / phase1)
  • 還原 legacy 功能、協議、配方格式、缺陷欄位
  • 盤點 gap(#1–#33)、決定遷移/won't-do
  • 寫實作計畫 + AskUserQuestion 裁示

實作 Coding

  • 遷移 code:kernel 本體 byte-exact 不改
  • 換 I/O 外殼、接線、補新增邏輯
  • 守不變式(DIV-only / network-clean / CCL 收斂…)
  • 同步寫 selftest / 驗證 CLI

驗證 Testing

  • Stage 1 合成影像驗算法正確性
  • Stage 2 端到端(offline-tcp / rdma-validate)
  • Stage 3 失敗路徑(錯誤輸入 → ERR 不 crash)
  • 貼數據 → 標 L-level → 更新 STATUS.md
考古的鐵律:kernel 不改11 個 GPU kernel 與 Demo 逐行 byte-identical。考古確認後只搬不改; 改 kernel 本體會破壞已驗證的 bit-exact 與跨架構一致性。

④ 驗證階梯:由 off-line 讀圖到接相機取像中心

系統不是一次到位接生產線,而是沿 5 步驟階梯往上,每階貼數據才進下一階。前段純軟體可在桌面跑,後段才上真實硬體。

1
off-line 讀圖驗證 — Control offline + IP offline-tcp,用 MIL 測試影像驗演算法。純軟體,桌面可跑
2
單 CCD 接相機 — Grab --cam-count 1 + IP rdma-validate,驗單台相機 → RDMA 傳輸。取像中心:1 台相機直連
3
全陣列 + Switch — Grab --cam-count ALL,37 台相機 + SN2201 交換器。待 Switch 到貨
4
存圖調 Recipe — full + 上位機 + IP image-capture,存 TIFF 回 Step 1 調參。閉環:offline = inline
5
完整生產 — full + 上位機 + IP online,接真實上位機跑線。
Step 4 → Step 1 閉環image-capture 存的影像回頭餵 Step 1 調 Recipe,保證「離線調的參數」= 「線上跑的結果」。 目前進度:Step 1–2 已驗(單相機 L4),Step 3 待 Switch,Step 4–5 為里程碑。

L0–L4 誠實分級(完成的定義)

「完成」= 驗證過,不是「寫完」。每個功能在 STATUS.md 標分級,標低不標高、有疑慮標保守。

級別意義判定
L0未實作沒程式碼或僅空殼
L1已寫程式碼邏輯寫完,未執行驗證(不得合入 production 路徑)
L2已單元自測selftest / unit test 通過
L3已端到端本系統內跑通完整流程(如 Mac↔Linux offline)
L4已與真實外部系統驗證接真實上位機 / 相機 / 硬體驗證過
L3 → L4 藏最多意外真實硬體/協議的邊界條件最容易爆。例:UpstreamServer(CF_/8787)程式寫好但從未接真機 → 誠實標 L1,不謊報 L4。

考古驅動遷移原則

遷移,不重寫從 Reference 搬現有程式碼;GPU kernel byte-exact 不改,只換 I/O 外殼。
legacy 相容RecipeInfo.xml / ResultInfo.xml / 缺陷欄位沿用 legacy 名稱與格式,接舊上位機鏈相容。
network-clean跨機不共用檔案系統:配方 XML 內容、結果 JSON、缺陷小圖 PNG 全 over TCP。
決定性同影像兩跑 bit-exact(CCL 收斂迴圈 + canonical 排序);off-line 調的參數對 on-line 有效。
不留幽靈號gap 盤點寧多列勿漏;考古無定義的舊編號標「無對應」,不硬湊。

一次人 × Claude 協作迭代

實務上,功能就是這樣一圈一圈收斂的:

Claude 交付 code + 文件 → 人在實機跑、貼數據 → 達標標 L3/L4 寫進 STATUS.md;不達標就帶著數據/bug 回到 Claude 再一圈。 多數本專案 commit(見 git log)就是這個循環的一次次落地。
