系統定位

系統把舊版單機 Windows(WinForms + MIL + GPU)拆成三台機器分工的分散式架構: 上位機下命令、Control 協調、Grab 取像、IP 在 GPU 上跑缺陷檢測。核心策略是從 Reference/ 遷移既有演算法(kernel 一字不改),只換 I/O 外殼。

IP

Linux / CUDA(→ DGX Spark GB10)。GPU 缺陷檢測引擎:8 方向鄰域 DIV 比例式閾值 + CCL + Blob。

Control

C# / Avalonia(Mac·Win·Linux)。控制平面 + 操作 UI;接上位機、協調 Grab/IP。

Grab

Linux / C++。相機擷取(pylon/eBUS)→ FrameHeader + CRC32 → RDMA 送 IP。

三機架構

上位機(另一台 PC,非 PLC)以文字命令驅動 Control;Control 對 Grab/IP 走 JSON;Grab 對 IP 走 RDMA 傳影像。

端到端數據流 — 一塊 panel 的生命週期

從上位機載入配方到回傳結果,一塊面板走過的完整路徑:

① 上位機 → Control:CF_LOAD_RECIPE
recipe / panelId / detectMode(8787,9 參數)
Control 讀 RecipeInfo.xml
② Control → IP:LOAD_RECIPE(JSON 8200)
送 XML 內容(非路徑,network-clean);IP 轉 ZoneConfig,DIV-only 強制
CF_GRAB_START
③ Control → Grab:GRAB_START(JSON 8100)
Grab 開相機,狀態 IDLE → GRABBING
RDMA WRITE_IMM
④ Grab → IP:影像幀(RDMA RoCE v2)
FrameHeader(magic 0xA01CF00D)+ Mono8 payload + CRC32;N-slot ring + credit 背壓
收圖驗證 magic/CRC
⑤ IP GPU 檢測
8-way DIV → CCL 收斂 → Blob → (AI 停用)→ canonical 排序 → InspectionResult
result_saver 落地
⑥ 輸出:ResultInfo.json/.xml + Defect_*.png + overlay
<output>/<yyyyMMdd>/<panelId>_<recipe>/
CF_GET_RESULT
⑦ 上位機 ← Control:結果路徑 + 缺陷數
OK/NG 判定:DefectCnt==0 即 PASS
network-clean(跨模組不變式) Control↔IP 跨機不共用檔案系統:配方傳 XML 內容、結果 JSON 與缺陷小圖 PNG bytes 皆 over TCP 回傳。 Grab↔IP 影像走 RDMA。

模組導航

ip/ · CUDA / C++

IP — GPU 檢測引擎

12 階段檢測 pipeline、DIV 比例式 kernel、CCL 收斂迴圈與決定性、RecipeInfo→ZoneConfig、結果輸出。

L4 演算法/CCL/跨架構於 GB10 實機驗證

 control/ · C# Avalonia

Control — 控制平面 + UI

5 個 UI 原型、RecipeStore 單一資料來源、上位機 CF_ 協議、DefectSort network-clean 分類。

L1–L3 混合;UpstreamServer 待接真機

 grab/ · C++ pylon

Grab — 相機擷取 + RDMA

取像→RDMA pipeline、N-slot ring + credit 背壓握手、FrameHeader 組裝、GB10 cudaHostAlloc。

L4 單相機 pylon→RDMA;多相機 L0

 tools/ · C++ OpenCV

tools — golden_maker

對位 Golden Pattern 製作(GUI 框選 / CLI),輸出 AlignRoi XML 片段,接 Gap #1 對位 pipeline。

L3 對位 pipeline 於 GB10 驗通
