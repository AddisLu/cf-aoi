Chapter 00 — Overview

這套系統在做什麼

CF-AOI 是彩色濾光片（Color Filter）面板的自動光學檢測系統：線掃相機掃過面板 → GPU 演算法找出亮/暗點缺陷 → 回報座標與大小給工廠上位機。這一章先建立全景，之後每章拆一塊。

一句話版本

Core Loop

相機拍 → Grab 收 → RDMA 傳 → IP 用 GPU 找缺陷 → 結果存檔並回報 → Control 指揮全場、上位機驗收。

Chapter 01 — Data Flow

一塊面板的一生：資料流

把「一塊面板從進站到判定 OK/NG」走一遍，你就掌握了整個系統的骨架。以下是生產模式（Step 5）的完整旅程。

① 上位機下令

CF_LOAD_RECIPE|T550QVN10|00201B0A|2026-07-11-08-30-00|||||||1\r\n
CF_GRAB_START|40000\r\n

上位機用純文字協議（port 8787）對 Control 下令：載入配方、開始取像（timeout 40 秒）。回應固定 9 參數：OK|p1|…|p8|errMsg 或 ERR|…。

② Control 分派

Control 把命令翻成 JSON：對 IP（port 8200）送 LOAD_RECIPE——注意配方是把 XML 內容整包傳過去（recipe_xml 欄位），不是傳路徑，因為 Control（Mac）和 IP（Linux）不共用檔案系統。這叫 network-clean 原則。對 Grab（port 8100）送取像命令。

③ 相機 → Grab → RDMA

線掃相機（Basler raL8192，8192 寬）隨面板移動連續掃出多張 slice。Grab 對每張：

  1. 組 FrameHeader（256 bytes：magic 0xA01CF00D、frameSeq、camId、sliceIndex、寬高、payload CRC32）。
  2. 經 RDMA WRITE_WITH_IMM 直接寫進 IP 預先註冊好的記憶體 slot（slot = frameSeq % N），100G 網路、單趟延遲 ~1.4µs。

④ IP：GPU 檢測

  1. 收圖入口驗證 magic / version / 尺寸 / CRC32——任一失敗記 incident 並拒收。
  2. 幀進 FrameQueue（固定上限，滿了就背壓，絕不動態擴記憶體）。
  3. GPU pipeline：前處理（依配方：Remap 對比拉伸、高斯平滑）→ 缺陷偵測 kernel（SUB 投票 / DIV 比例，見 Ch4）→ CCL 連通元件 → Blob 統計與過濾 → canonical 排序。
  4. 依配方 ROI 與對位位移（eff_*）算出每顆缺陷的全域座標。

⑤ 結果落地與回報

<output>/20260711/00201B0A_T550QVN10/
   Defect_IP04_Slice12_Roi0_Run00_X5894_Y135725_DrDark.png   ← 缺陷小圖
   00201B0A_T550QVN10_ResultInfo.json                        ← Control 讀
   00201B0A_T550QVN10_ResultInfo.xml                         ← 上位機鏈相容
   00201B0A_T550QVN10_result.png                             ← overlay 大圖
<output>/_diag/20260711.jsonl                               ← 行車紀錄（出事才有）

結果雙寫：JSON 給 Control 反序列化、XML 走 legacy JudgeResult 格式讓上位機的 CF_GET_RESULT 鏈相容。判定規則極簡：DefectCnt == 0 即 PASS。

⑥ 上位機收尾

CF_GET_RESULT|...  →  OK|IP04_00201B0A_T550QVN10,…|1,…|…

回的是「各 IP 的 ResultInfo.xml 路徑 + 缺陷數」（逗號分隔的文字），不是 JSON——這是考古自 legacy 的既定格式，不能改。

閉環設計

Step 4 存的圖 = Step 1 調參的圖。image-capture 模式存下的 TIFF 拿回離線模式調配方，保證「離線調出來的參數，上線行為一致」（offline = inline）。這是整個 5 步驟驗證流程的核心保證。

Chapter 02 — Protocols

通訊協議三層

系統有三種完全不同的通訊協議，各有歷史原因。搞混任何一層，兩端就對不上話。

第一層：上位機 ↔ Control（CF_ 文字協議，port 8787）

格式：CF_命令|參數1|參數2|…\r\n，回應固定 9 參數：OK|p1|…|p8|errMsg 或 ERR|…

命令範例 / 說明
CF_LOAD_RECIPECF_LOAD_RECIPE|{recipe}|{panelId}|{時間戳}|||||||{detectMode}
CF_GRAB_STARTCF_GRAB_START|40000（timeout ms）
CF_CHECK_ALIGN / CF_SET_ALIGN對位檢查 / 套用位移 |{result}|{shiftX}|{shiftY}
CF_GET_RESULT回「ResultInfo.xml 路徑 + 缺陷數」逗號串，非 JSON
CF_STOP中斷取像

實作在 control/src/Controllers/UpstreamServer.cs；這些命令名、前綴、9 參數格式是不變式——舊上位機程式已經寫死，改了就接不上。

設計決策 A：誠實失敗

離線環境沒接相機時，CF_GRAB_START/CF_CHECK_ALIGN/CF_SET_ALIGN 回 ERR，不回假 OK。寧可上位機看到錯誤，也不能讓它以為「已對位」。這個「不假 OK」原則全系統通用。

第二層：Control ↔ Grab / IP（newline JSON，port 8100 / 8200）

{"cmd":"LOAD_RECIPE","seq":1,"params":{"recipe":"DEFAULT","panel_id":"T001",
  "recipe_xml":"<Recipe>…</Recipe>", "recipe_saving":{…}, "share_flags":{…}}}
{"cmd":"CHECK_HEALTH","seq":3,"params":{}}   ← 心跳，連線燈綠/紅靠它

一行一個 JSON。重要命令族：

  • 檢測：LOAD_RECIPE（帶 XML 內容）、SEND_IMAGE_FOR_REVIEW（上傳影像+檢測）、REVIEW_LOCAL_IMAGE（IP 讀自己磁碟的圖檢測，結果與上傳同圖 bit-exact）。
  • DefectSort 遠端歸檔：LIST_DEFECT_FOLDERS / SORT_DEFECTS / LIST_DEFECT_PATCHES / GET_DEFECT_PATCHES_BATCH（小圖 base64 批次回傳）/ SAVE_DEFECT_CLASSIFICATION（人工標 TrueDefect/Particle）。
  • 相機（Grab 8100）：LIST_CAMERAS（MAC 列舉）、SET/GET_CAM_PARAMS（曝光/增益）、GET_CAM_NODES、TUNE_MEAN。
陷阱：UTF-8 逐 byte 解碼

C# 端收 TCP 必須累積整行 bytes 再解 UTF-8（IpClient）。逐 byte 轉 char 會把中文切爛——這是修過的真實 bug（缺陷分類中文亂碼）。

第三層：Grab ↔ IP（RDMA wire，FrameHeader 256B）

struct FrameHeader {            // 固定 64B 欄位 + 192B reserved = 256B
    uint32_t magic;             // 0xA01CF00D（舊 0xCFA0A001 作廢！）
    uint16_t version;           // 2
    uint16_t headerBytes;       // 256
    uint64_t frameSeq;          // 也塞進 RDMA imm
    uint32_t panelId;  uint16_t camId, sliceIndex, totalSlice, scanStep;
    uint32_t width, height;     // 8192 × 5000
    uint16_t bitDepth, pixelFormat;
    uint64_t ptpTimestampNs;
    int32_t  machineCoordX, machineCoordY;
    uint32_t payloadBytes, crc32;   // payload CRC32 (IEEE 0xEDB88320)
    uint8_t  reserved[192];
};  static_assert(sizeof(FrameHeader)==256);

兩端共用 shared/FrameHeader.h，版本必須一致、sizeof 必須是 256。這份格式是 2026-06-11 實機 RDMA 全幀 CRC 驗證過的版本。

network-clean：為什麼一切都走 TCP 傳內容

Control（Mac）↔ IP（Linux）跨機、無共用磁碟。所以：配方傳 XML 內容、結果傳 JSON、缺陷小圖傳 PNG bytes（base64）、對位 golden 傳 base64 PNG。任何「IP 去讀 Control 的檔案」或反過來的設計都是錯的。唯一例外是 REVIEW_LOCAL_IMAGE——那是 IP 讀自己磁碟上的圖（大圖搬不動，只回縮圖預覽）。

Chapter 03 — CUDA Primer

CUDA 入門：看懂 IP 的 GPU 世界

你不需要會寫 kernel 才能維護這套系統——但需要看得懂它在做什麼、為什麼有那些「禁改」規定。這章用本專案的實際程式碼當教材。

GPU 平行計算的三層結構

CPU 一次跑幾十個執行緒；GPU 一次跑幾萬個。CUDA 把它們組織成三層：

  • Thread（執行緒）：最小單位。在本專案裡，一個 thread 負責一個像素。
  • Block（區塊）：一組 threads，共享快速記憶體（shared memory）。本專案固定 16×16 = 256 threads/block（不變式：block_dim 寫死 16×16，INI 裡的 32×32 是從未生效的死設定）。
  • Grid（網格）：整張影像需要的所有 blocks。8192×5000 的影像 ≈ 512×313 個 blocks 同時排隊上 GPU。
// 一個 kernel（GPU 函式）長這樣：__global__ 表示「在 GPU 上跑」
__global__ void kernelDetect(const uint8_t* img, int w, int h, ...) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;   // 我是哪個像素？
    int y = blockIdx.y * blockDim.y + threadIdx.y;
    if (x >= w || y >= h) return;                    // 邊界防呆
    // …對 img[y*w+x] 這一個像素做判定…
}

本專案的 GPU 檢測流水線

  1. 上傳：影像從 host（CPU）記憶體進 GPU。生產機 GB10 是 SoC 統一記憶體，用 cudaHostAlloc(Portable|Mapped) 讓 GPU 直接讀 host 記憶體（NVLink-C2C ~900GB/s），不能用 nvidia_peermem（GB10 的 GPU 不在標準 PCIe 空間，實機驗證會 EINVAL）。
  2. 前處理（依配方）：Remap 對比拉伸（min/max 線性拉滿動態範圍）、3×3 高斯平滑。
  3. 缺陷偵測 kernel：每像素跟「週期性鄰居」比較（見下節）。輸出一張 defect mask。
  4. CCL（Connected Component Labeling，連通元件標記）：把相鄰的缺陷像素合併成「一顆缺陷」。
  5. Blob 統計 + 過濾：算每顆缺陷的面積/寬高/位置，按 size 過濾、鄰近合併。
  6. 下載 + canonical 排序：結果回 CPU，依 label → center_y → center_x → size 排序。

為什麼跟「週期性鄰居」比？

CF 面板是規則的 RGB 網格，週期 = PitchX / PitchY。正常像素和「一個 pitch 之外的同色像素」幾乎一樣亮；缺陷則不一樣。所以演算法不是「找黑點」，而是「找跟自己的週期鄰居不一致的點」。

爆量陷阱（血淚不變式 14）

Pitch 設錯 1–4 px，整片正常網格都會被判成缺陷：實測 PitchX 26→30，缺陷數 561→10000（觸頂 MAX_DEFECTS）。新面板一定先用 FFT 估算 Pitch（scripts/estimate_pitch.py 或 Step1 的 FFT 鈕），不可沿用舊值或用猜的。缺陷數接近 10000 時，第一個懷疑對象就是 Pitch。

兩套偵測演算法：DIV 與 SUB

DIV（比例式，Demo 移植）SUB 投票式（legacy 移植，生產配方用）
比較方式中心 ÷ 8 鄰平均，比值 vs BTH/DTH中心 − 每個週期鄰居，逐路灰階差
閾值語意比例（如 1.30 / 0.75）灰階差（如 +17 / −16）
判定單次比較投票：PitchTimes×8 路（如 16 路）中 ≥ ChooseAmount（如 13）路超標才算缺陷
容錯—每路先做 3×3 SAD 局部搜尋，容忍面板 skew
重要歷史轉折

系統原本是 DIV-only、SUB 拒載。2026-06 底發現真實生產配方（T550QVN10 全 24 顆 CCD）全是 SUB，且守門曾被配方裡殘留的過期 <AlgorithmCompare>DIV</AlgorithmCompare> 字串騙過、靜默假 PASS。於是決策移植 legacy SUB 投票管線（docs/SUB_pipeline_port_plan.md），守門改讀 M_AlgorithmWayCompare。教訓：用字串比對做的守門，必須驗證讀的是「真正決定行為的欄位」。

決定性（bit-exact）：為什麼是鐵律

不變式：同一張圖跑兩次，結果每個 byte 都相同。否則調參就沒有意義（你不知道差異來自參數還是隨機性）。GPU 天生會破壞這件事，本專案修了兩處：

  • CCL 收斂迴圈：原版 lock-free union-find 只跑一次、受 thread race 影響（缺陷數會 2761/2762/2763 飄）。修法是 host 端迴圈呼叫 merge kernel 直到 d_changed==0——收斂後的不動點唯一。⚠️ 若有人從 Reference 重新複製 kernel 檔，這個迴圈會被蓋掉，決定性立刻壞。
  • Canonical 排序：blob 結果用 atomicAdd append，順序隨 race 變動（集合相同、順序不定），下載後必須排序成固定順序。
禁改規定的由來

cuda_kernels.cu / ai_kernels.cu 的 __global__ kernel 本體禁止修改——它們是從已驗證的 Reference/Demo 直接複製的。可以改的是 host 端的「編排」（launch wrapper、收斂迴圈、排序）。新演算法（如 SUB 投票）只能新增 kernel，不動舊的。

x86 開發 ↔ ARM 生產：跨架構一致性

家裡 RTX 2080（x86, sm_75）調的參數，拿到 Spark（ARM GB10, sm_121）有效嗎?——實機驗過：26 張真實面板 25/26 缺陷數完全一致；整數/幾何欄位完全一致，唯浮點閾值邊界像素可能單像素 ULP 翻面，方向偏保守（寧抓勿漏），該缺陷會進 DefectSort 人工複核。結論：off-line 調參對 on-line 有效。

Chapter 04 — Recipe

配方 Recipe：系統的靈魂

配方（RecipeInfo.xml）決定「檢什麼、怎麼檢、多嚴格」。它是 legacy 相容格式——欄位名一個都不能改，因為舊系統和舊配方檔都在用。

└── Recipe

命名考古

舊文件說閾值叫 ThB/ThD——錯，那是 GPU 端 CUDAZone 結構的內部欄位名。XML 裡永遠是 BrightThreshold/DarkThreshold，由 ThB=(float)BrightThreshold 直接賦值。同理「ZoneSetting」也是誤傳，實體是 DetectRoi。寫程式前先考古，別信二手文件。

StartX/Y, EndX/Yzone 範圍-1 = 全幅

結果格式：ResultInfo（也是 legacy 相容）

缺陷欄位一律用 legacy 名稱：GC_X/GC_Y（全域座標）、Size、Width/Height、X_Min/X_Max/Y_Min/Y_Max、Type（PointBright/PointDark）、GL_Mean、GlobalPosX/GlobalPosY（另有 _um 微米版，單位契約待上位機確認）。GPU 沒有的欄位（CV_*、*_Sigma）填 0。結構：JudgeResult → RoiInfoList → RoiInfo → DefectInfoList → DefectInfo。

對位（Alignment）：面板放歪了怎麼辦

  1. 每片面板取像後，Control 下 CHECK_ALIGN：IP 用 OpenCV pattern match（TM_CCOEFF_NORMED + 拋物線 sub-pixel）在搜尋窗裡找對位 Mark，回 ShiftX/ShiftY（實測精度 <0.1px）。
  2. SET_ALIGN 把位移套回所有 zone：aligned_* = roi_* + round(shift)；偵測統一走 eff_*()（有對位值用對位值，否則用原 ROI）。
  3. 分數低於門檻 → 回 ERR（不回錯誤位移），由上位機決策停線/放行——又是「誠實失敗」原則。
已修的真實 bug（F1）

全幅 zone（StartX=-1）套對位位移後 -1+7=6，不再被認出是全幅 → zone 縮成 1px、檢測範圍塌掉。修法：全幅 zone 跳過位移套用。教訓：魔術值（-1）跟算術運算混用時，每個運算點都要先判魔術值。

配方管理紀律

  • 單一資料來源：Control 端三處 UI（Step1 快速調參、ZoneParamEditor、單 CCD 工作台）共用同一個 RecipeStore 實例——改一處、處處同步，存檔才寫回 XML。
  • per-CCD 隔離：切換 CCD（IP0→IP1）會重載該分區的 XML；IP0 改的參數不會滲到 IP1。
  • 用真配方，勿手寫假設值：SUB 移植時曾因手寫測試配方誤用別顆 CCD 的 ROI，追了很久的「漏檢」其實是 ROI 排除了缺陷。
Chapter 05 — RDMA & Backpressure

RDMA 與背壓：影像怎麼跑 100G

一張 8192×5000 的 Mono8 影像約 39MB；37 顆 CCD 一塊面板約 1110 張。普通 TCP 搬不動，所以 Grab→IP 走 RDMA——網卡直接寫對方記憶體，不經 CPU。

RDMA 基本觀念（給沒碰過的人）

  • RDMA = Remote Direct Memory Access：送端網卡把資料直接 DMA 進收端預先註冊好的記憶體區（MR），收端 CPU 全程不參與搬運。延遲實測 ~1.4µs。
  • RoCE v2 = RDMA over Converged Ethernet：在一般乙太網上跑 RDMA（本系統用 100G DAC 直連，link_layer 必須是 Ethernet）。
  • WRITE_WITH_IMM：寫入之外附帶一個 32-bit 立即值（本系統塞 frameSeq），收端收到完成事件才知道「有一幀到了」。
  • QP / CQ：Queue Pair（收發佇列）與 Completion Queue（完成通知）。QP 進 error state = 連線壞掉。

N-slot ring：收端記憶體怎麼配

IP 端啟動時：cudaHostAlloc N 個 slot（預設 4 × 40MB pinned host memory）
握手：IP 把 MrInfoEx{addr, rkey, n_slots, slot_size} 傳給 Grab
Grab 送第 seq 幀：寫進 slot = seq % n_slots
IP recv_thread 收到完成事件：
   [1] memcpy slot → 自己的 buffer
   [2] push_blocking(FrameQueue)      ← 隊滿就在這裡等
   [3] post_recv                       ← 補一張「收據」(credit)
背壓鏈（不變式 23）

1-2-3 的順序不可調換。post_recv 一定在 push_blocking 之後——保證 Grab 在 IP 讀完 slot 前拿不到新 credit、不可能覆寫。整條鏈：GPU 慢 → FrameQueue 滿 → push_blocking 卡住 → 不補 credit → Grab 端 RNR → poll_one 阻塞。Grab 自然降速而不斷線（實測 IP 端 200ms 延遲 → Grab 從 1375fps 降到 9.6fps，QP 不進 error state）。

三個實機踩過的坑

坑 1：33fps 牆（min_rnr_timer）

credit 短暫耗盡時，送端收到 RNR NAK 後會等 min_rnr_timer 再重試——rdma_cm 預設值是 655.36ms！吞吐直接崩到 2.6fps。更陰險的是：舊版每幀多花 16ms 算 app-CRC，「意外」把送端降速到跟收端一致、剛好避開 RNR；一關 CRC 送端變快反而觸發崩潰。修法：收端 accept 後 ibv_modify_qp(min_rnr_timer=12)（0.64ms）。結果 33 → 73.8fps。教訓：效能問題的「原因」可能是別處的「減速帶」被移走。

坑 2：RoCE 斷線偵測不可靠

RoCE v2 上 Grab 斷線後 WR_FLUSH_ERR 不保證出現 → recv_thread 永不退出 → FrameQueue 永不 close → 主迴圈永久卡死。修法：no-event 分支輪詢 CM event channel（check_cm_disconnect()）。教訓：分散式系統裡「對方死了」的訊號永遠要有第二來源。

坑 3：GB10 不能用 nvidia_peermem

DGX Spark 是 NVLink-C2C SoC，GPU Bus ID 不在標準 PCIe 空間，modprobe nvidia_peermem 直接 EINVAL。正解：cudaHostAlloc(Portable|Mapped) pinned host memory 註冊 MR，GPU 經 cudaHostGetDevicePointer 走 NVLink-C2C（~900GB/s）讀。教訓：換硬體平台時，「業界標準做法」要重新驗證。

記憶體紀律：固定上限，絕不動態擴

legacy 系統曾因「每幀 new 一個 buffer 塞進 List」在生產環境 OOM。新系統的鐵律（不變式 18/19）：

  • FrameQueue：啟動時算一次上限（50% 可用 host RAM ÷ 幀大小，至多 8 幀），滿了回 ERR「系統繁忙」+ 記 incident，絕不配更多記憶體。
  • SourceImageWriter（存原圖）：固定 N 個 ring slot + 獨立寫檔執行緒，ring 滿就 drop + 記 incident，絕不阻塞主路徑、絕不囤 List。
  • host RAM 用 sysinfo().freeram、GPU RAM 用 cudaMemGetInfo——兩者絕不混用。
Chapter 06 — Flight Recorder & Runbook

機制：環形緩衝 + 出事才落地

  • 平時：最近 64 張幀的現場（panel、zone 參數、佇列水位、GPU 狀態…）存在記憶體環形緩衝，零磁碟 I/O。

Chapter 07 — Engineering Discipline

工程紀律：L0–L4 與不變式文化

這個 repo 有兩個特別的文化制度：誠實分級（L0–L4）和不變式清單。理解它們，你就知道「哪些話能信、哪些線不能踩」。

L0–L4 誠實分級：「寫好」不等於「驗證過」

級別意義判定標準
L0未實作沒有程式碼，或僅佔位
L1已寫程式碼邏輯寫完，未執行驗證——不得進生產路徑
L2已單元自測selftest / unit test 通過
L3已端到端驗證本系統內跑通完整流程並貼出數據
L4已與真實外部系統驗證真上位機 / 真相機 / 真硬體
  • 規則：標低不標高；沒貼數據就不算完成；L3→L4 之間藏最多意外。
  • docs/STATUS.md 是全系統的誠實帳本——接手第一件事就是讀它，分清「真的能用」和「只是寫了」。
  • 範例：AI 分類器「模型有載入」= L1（預設停用、不推論，缺陷全標「待人工複核」交 DefectSort 人工分類）。

驗證三階段（每階段貼數據再往下）

  1. Stage 1 合成影像：已知答案的人造圖驗演算法正確性（不需相機）。
  2. Stage 2 端到端：offline-tcp / rdma-validate 跑通，貼缺陷數、幀數、CRC、誤差。
  3. Stage 3 失敗路徑：餵錯誤輸入 → 必須正確回 ERR，不 crash、不 silent fail。
