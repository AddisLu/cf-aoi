① 座標轉換該怎麼做（契約）

本系統用 模型 A（Legacy 原汁）：每台 CCD 各自一份本地像素 ROI，加上各自的對位 Mark（M_AlignRoi）來吸收面板放置的起始點差異。沒有「全域 ROI」這回事——全部以該 CCD 影像的像素座標表示。

座標系定義

  • 原點＝該 CCD 影像左上角 (0,0)；X 向右增、Y 向下增（影像像素座標）。
  • ROI 用 StartX / StartY / EndX / EndY 表矩形；任一邊 -1 = 全幅。
  • 對位後缺陷的全域座標 GlobalPosX/Y 仍是像素；可再乘光學解析度得 μm。

完整轉換鏈（教導 → 執行 → 套回 → 缺陷座標）

階段做法公式
1. 教導在影像上標 Mark 期望位置 ReferX/ReferY，截下該處 golden 樣板（PatternPath）。Refer = 「Mark 應該在哪」的基準點
2. 裁搜尋窗執行時從實拍幀以 Refer 為中心裁一塊 SearchWidth×SearchHeight。offset = (ReferX − SearchW/2, ReferY − SearchH/2)
3. 找 Mark在搜尋窗內做 pattern matching（歸一化相關），取最高分位置（+ 次像素擬合）。得 mark 在搜尋窗內的中心 (markCx, markCy)
4. 算位移Mark 實際中心 − 搜尋窗中心 = 偏移量。ShiftX = markCx − SearchW/2
ShiftY = markCy − SearchH/2
5. 套回 ROI所有 DetectRoi 整體平移 Shift（純平移，不旋轉）。AlignedStart = Start + Shift
AlignedEnd  = End  + Shift
6. 缺陷全域座標ROI 內局部質心 + 該 ROI 在影像中的偏移。GlobalPos = roi_offset + 局部質心
μm = GlobalPos × opt_res

🔑 樞紐契約（最關鍵一句）

位移計算的本質是 「Mark − 搜尋窗中心」。因此這個位移要等於「Mark 相對教導點 Refer 的偏移」，唯一前提是搜尋窗必須恰好以 Refer 為中心裁切。一旦搜尋窗中心 ≠ Refer（例如近邊界被夾小），Shift 的物理意義就跑掉了。新舊版都靠這條契約成立。

🔴 F1（真實 bug）：對「全幅 zone」套對位 → ROI 被毀成 ~1px

ip/src/main.cpp:462-467 無條件對所有 zone 算 aligned_* = roi_* + round(shift)。當 zone 為全幅（roi_start_x = -1）且 shift > 0：

aligned_start_x = -1 + 7 = 6 (≥0)  →  is_full_frame() 變 false  →  zone_rect 算出 x1=6, x2=7 = 1px 寬

整張全幅檢測區被縮成 1 像素。負 shift 矇對、混合 shift 更亂 → 非決定性錯誤。Legacy 無此問題（其全幅是 -1 在演算法層另行處理，未做 -1 + shift 的整數運算覆蓋）。

修法：SET_ALIGN handler 先跳過全幅 zone，保留 -1 sentinel：

for (auto& z : zones) {
  if (z.is_full_frame()) continue;   // 全幅不套位移
  z.aligned_start_x = z.roi_start_x + round(shift_x);
  ...
}

🟠 F2（契約風險）：搜尋窗必須以 Refer 為中心，且近邊不可夾小

新版 shift = mark − search_roi 中心，所以 Control 端裁搜尋窗時必須恰為 SearchW×SearchH、中心 = ReferX/Y。但 IpClient.CheckAlignAsync 只收「已裁好」的 payload，真正從實拍幀做中心裁切並呼叫它的生產程式尚未實作（offline 回 ERR）。Legacy 的 checkBoundary 近邊會夾小搜尋窗 → 中心偏移 → shift 語意跑掉（legacy 隱性靠 Refer 遠離邊界）。

動作：把契約寫成不變式 + 文件（「搜尋窗中心 = Refer；近邊以補零維持尺寸，不得 clamp 縮小」）。實際裁切串接屬 Step4 取像，本次不實作。

🟡 F3（已知限制）：旋轉只回報不套用，且範圍僅 ±3°

angle_deg 有偵測但 SET_ALIGN 只套平移（與 legacy 同，model A）。單 CCD 小 FOV + 各自 Mark 下殘留旋轉誤差小，可接受；但 legacy ±180°、新版僅 ±3°，若面板擺放角度偏大會匹配失敗。動作：文件標明邊界，並確認 score_threshold 能擋大角度誤匹配。

⑥ 掃描幾何：一顆 CCD 的多張影像怎麼拼、怎麼對位

線掃描下，一顆 CCD 的一塊面板 ＝ 多張 frame 垂直疊成「一張連續大圖」（高 = ImageSizeY × FrameCnt）。對位和 ROI 都在這張大圖的像素座標上做 —— 不是每張各自對位。

同一顆 CCD — 上下（frame 之間）

frame 貼合堆疊、零 overlap（childOffsetY = ImageSizeY × frame）。Mark 在整片只比一次 → 一個位移，所有 frame 共用。

相鄰 CCD — 左右（CCD 之間）

各 CCD 獨立影像、獨立 recipe；左右重疊 / 不可用邊靠 IndentLeft/IndentRight 內縮裁掉，不靠 overlap。缺陷最後由上位機 CF_GET_RESULT 依全域機器座標合併。

運算時機與輸出合併（online 生產流程）

① 對位計算 → ② 收一張算一張（每張套同一平移，不套旋轉）→ ③ 輸出合併
面向做法出處
對位拼接大圖 Mark 比一次 → ShiftX/Y（純平移）；旋轉量到了卻沒套用（tmp_PatAngle 抽出後沒人用）CamProc.cs:395-445
運算時機inline 生產：對位後收一張算一張（每收一張就偵測該張，不等整片）；offline 調參：整片載入記憶體後再算CamProc.cs:1401-1419

⑦ 新對位方法提案：兩 Mark / Golden 雷射

現用法 ＝ 單一 Mark → 純平移（量得到旋轉但丟掉）。兩種升級方案本質相同：多點 → 解出剛體變換（平移 + 旋轉），差別只在「點怎麼來」。

Method 1 — 第一張 + 最後張 兩 Mark

兩點沿掃描軸拉出長基線 → θ 量得夠準。改動較小，但無雷射真值、仍靠手動 teach。可視為 Method 2 的 2 點特例。

Method 2 — Golden Sample 雷射打點（N 點真值座標）

最穩、免去手動 teach 的誤差：雷射打點的座標就是已知真值（ground-truth），多點互相備援、更可靠。並可順便校正 pixel↔μm（opt_res）。需要一份雷射座標校準檔。

🔑 關鍵岔路：量旋轉「容易」，套旋轉「難」

兩法都能算出旋轉 θ（OpenCV 幾行：2 點剛體 / N 點 estimateAffinePartial2D）。真正的成本在「怎麼把 θ 套用到偵測」—— 它撞到兩道現有約束：

① kernel 假設「網格軸對齊」：8-way DIV 用 pitch_x/pitch_y 取鄰點。面板一轉 θ，影像裡的週期網格也轉 → 鄰點取到錯相位 → 整片假缺陷。只轉 ROI 角點、bounding box 軸對齊沒用（框內 pattern 還是斜的）。

② 要真校正只有一條路：把整張影像反旋轉 θ（幾何重取樣 warp）回正再偵測。代價：插值 → 破壞 bit-exact（同圖兩跑結果逐位元相同的保證）（不變式 #10/#12）+ 模糊次像素缺陷 + warp 成本 + offline 調的參數要在「回正影像」上重新驗一次基準。

ℹ️ 什麼是 bit-exact？

bit-exact 是指同一張影像、用同一份程式跑兩次（或在不同機器上跑），輸出的每個 bit 都要完全一樣，連一個像素的值都不能差。這專案在意它，是因為缺陷檢測的參數是在 off-line（x86）調好的、上線要在另一台機器（ARM GB10）跑生產；只有結果逐位元一致，off-line 調的參數才保證對 on-line 同樣有效，不會無端多抓或漏抓。一旦做了影像反旋轉 warp 這種插值，就會破壞這個保證，等於要重新驗一次決定性、重調參數基準。

落地策略：兩條路（先決定再寫碼）

策略做法代價 / 風險適用
A（建議） 只套平移 + 旋轉超標即退件（reject）多點只用來把平移量算穩 + 多一道「面板擺歪」健康閘（|θ| > 容差 → ERR）不動 kernel、保 bit-exact、最省面板殘留旋轉小（單 CCD 小 FOV）
B 真校正旋轉加整張反旋轉 warp 前處理回正，再跑現有軸對齊 pipeline破 bit-exact、需重驗決定性 + 重調參數基準、warp 成本真實面板會轉超過容差時才值得
