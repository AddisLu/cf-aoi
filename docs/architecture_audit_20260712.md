# CF-AOI 全系統架構審計報告（2026-07-12）

> **方法論**：多代理審計 workflow——6 位讀者平行測繪（ip/grab/control/協議/SUB移植/測試資產）→ 7 個風險維度獵人（傳輸/併發/契約/錯誤路徑/規模/營運）→ 每個 finding 由獨立代理**對抗式驗證**（設法推翻，親讀源碼）→ 行車紀錄器與預相機測試兩個專項評估。共 ~60 個代理、~970 次工具呼叫、~5.2M tokens。
> **驗證狀態標記**：✅=對抗驗證確認（獨立代理重讀源碼）、🔷=多代理交叉確認+人工抽查（未跑完整對抗驗證）、⚠️=單一來源、待驗。
> **對照基準**：commit `fe5cc9b`（2026-07-11 時點 main）。file:line 為審計時實際讀到，後續 commit 可能位移。

## 執行摘要

**架構本身成立**：三程式分層（Control 控制面 / Grab 取像 / IP 運算）、三層協議（CF_ 8787 / JSON 8100·8200 / RDMA FrameHeader）、network-clean、RDMA credit 背壓鏈、bit-exact 決定性文化、L0–L4 誠實分級——這些骨架設計經逐檔核實**與文件一致且合理**。實機驗證過的部分（單相機 RDMA 全鏈、對位、DefectSort、SUB 核心移植）品質紮實。

**但有兩個「上線即出事」級的硬結論**：

1. **產能結論已失效**（✅）：「1 台 Spark 足夠、餘裕 73%」建立在舊 DIV 演算法 7.4ms/張；SUB 真配方在 GB10 實測 **24 CCD GPU 合計 ~40.3s，已超 30s 節拍**（37 CCD 外推 ~62s）。全 repo（docs/CLAUDE.md、STATUS.md、Control UI 負載估算、bench 換算公式）無任何更新後的投影。
2. **生產路徑缺 sliceIndex→Y-offset 映射**（✅）：SUB 全 panel 配方（ROI Y 可達 146k）目前只有 `offline-file --stitch` 能正確跑；逐 slice 的 offline-tcp/rdma-process 路徑會把超幀 ROI **靜默 clamp 成 1px 條帶**→ 近 0 缺陷假 PASS（`ip/src/main.cpp:177`，`ip/src` 全域 grep `sliceIndex` 零消費）。這是「stale-DIV 假 PASS」事故的座標軸版重演，且**不需相機就能修＋驗**（原 Step B 規格＋回放鏈）。

**第三個系統性主題**：**「靜默假 OK」家族**在跨機邊界大量存在（CF_GET_RESULT 永遠回 OK、配方名打錯自動生成預設配方、Grab 送端斷線後靜默丟幀且健康檢查照綠、mode2 閾值域不防呆）。單機內的誠實失敗文化（決策 A）沒有貫穿到跨機組合層。

**第四個主題**：**文件大規模飄移**。SUB/融合移植（12 個 commit）推翻了「DIV-only」不變式，但 docs/CLAUDE.md §5/§7、ip/CLAUDE.md 不變式 9、STATUS.md、zone_config_adapter.h 檔頭註解全部還寫舊規則——這正是上次假 PASS 事故的溫床（文件 ≠ 程式碼時，下一個人照文件做事就出錯）。

## 一、現況架構確認（審計後修正版）

### IP 偵測管線現況（文件未載，以此為準）

`algo_mode` 三分支（守門 `zone_config_adapter.cpp:114-211`，權威欄位 **M_AlgorithmWayCompare**）：

| mode | 演算法 | 閾值語意 | 前處理 | 多尺度/LSC |
|------|--------|---------|--------|-----------|
| 0 | DIV（舊 Demo 比例式 8-way） | 比例（1.40/0.60）| 無 | 不接 |
| 1 | SUB（legacy 8-Way-Star：16 路逐路灰階差＋3×3 SAD＋ChooseAmount 投票）| 灰階差（+17/−16）| Ip_Remap＋Gau3×3 平滑 | 不接 |
| 2 | DIV-voting（融合：比值逐路投票＋暗區棄權＋比例門檻）| 比例 | 僅平滑（remap 刻意不跑）| **僅此模式**接多尺度 2×/4× OR-merge＋LSC |

- 守門判定：含 `Sub`→mode1；含 `div`+`star/way`→mode2；否則 fallback 舊 `AlgorithmCompare=DIV` 字串→mode0（DTH<0 防呆拒載僅在 mode0 分支）。
- GPU 之後的 CPU 後處理鏈：`defect_rules::apply_blob`（Step E size 過濾＋union-find 鄰近合併）→ #32 邊界略過＋#16 Rule 改判 → MaxDefectCountPass 截斷。
- 生產配方（T550QVN10 全 24 顆）= `Awc_8_Way_Star_Sub` → mode1。已驗收：IP04 真配方全 31 slice 拼接，027/028 已知缺陷正確檢出、全 panel 46 缺陷。

### 模式清單（實際存在，文件缺列）
`offline-file`（含 `--stitch`）、`offline-tcp`、`bench`、`rdma-validate`、`rdma-process`（RDMA→完整 AOI＋塞車監控——事實上的準 online 路徑）。`image-capture`/`online` 正式模式仍 L0。

## 三、行車紀錄器實戰有效性評估（Q2）

**一句話結論**：對「IP 行程內 crash」很強（可單檔定位），對使用者最怕的三類生產 bug 幾乎全盲——**結果錯但不 crash、整線 hang、Grab/Control 端問題**。

### Bug 類型 × 可診斷性矩陣

| Bug 類型 | 評分 | 現況 | 缺口 |
|---------|------|------|------|
| IP crash / CUDA fatal | **可定位** | cuda_fatal/uncaught_exception 帶最近 64 張現場＋GPU 記憶體＋complete 旗標 | SIGSEGV/OOM-kill 不觸發 atexit → 零紀錄；無 stack |
| **結果錯不 crash（爆量/漏檢）** | **全盲** | ring 正常結束即蒸發，無任何異常觸發器 | 觸頂 cap、pass 率驟變、零缺陷連續 N 張都不產生 incident；**ZoneSnap 只記 DIV 時代 8 欄**——algo_mode/multiscale/blob/LSC 全不在快照，「這張用哪個演算法跑的」答不出 |
| 整線 hang（背壓卡死）| TCP 部分 / RDMA 盲 | TCP 有 queue_overflow/high_watermark | RDMA 背壓鏈零 incident（main.cpp:674 註解承諾的 credit incident 未實作）；無 watchdog——程式不退出就永遠沒 incident |
| RDMA 掉幀/CRC | 部分 | 收端 magic/CRC 錯有 incident | **送端（Grab）失敗靜默丟幀無任何紀錄**——「Grab 覺得有送、IP 沒收到」兩端都無證據 |
| 配方載錯/靜默預設 | 部分偏盲 | 解析拋例外有 recipe_load | **LOAD_RECIPE 成功零落盤**——守門 mode 判定只在 stdout；「載錯但合法」整族無痕跡 |
| Grab 端（相機掉線/曝光）| **盲**（無行車紀錄器）| printf 而已，重啟即失 | — |
| Control 端（命令沒送）| **盲**（無持久 log）| — | — |
| 跨機問題 | 弱 | panel_id+frame_seq 可弱關聯 | 只有 IP 落盤；無 trace id；TCP 路徑 timestamp=0 |
| 效能退化 | 盲偏部分 | gpu_ms 在 ring 不落地 | 無週期 stats → 「上週 7.4ms 這週 12ms」無資料可比 |

### 半夜排障現實
從「上位機報 NG 異常」出發：要連 **3 台機器**、開 2–4 種檔案，其中 damac/Mac 只有可能已捲掉的 console（repo 無 systemd/日誌輪替）。唯一能單檔看到 root cause 的劇本是「IP 行程內 crash」。行車紀錄器目前回答「IP 怎麼死的」，回答不了「系統為什麼算錯/為什麼停」。

### 改進建議（投資報酬排序）

**T1（各 <50 行，先做——把最危險盲區從全盲拉到可定位）**
1. ZoneSnap 補 SUB/融合欄位（algo_mode/multiscale/pitch_times/choose_amount/blob_*/lsc/preproc）。
2. LOAD_RECIPE **成功**也寫一行 jsonl（recipe 名/XML hash/每 zone 守門判定）→ 封殺「守門誤分類無痕跡」整族。
3. `defect_flood` 觸發器：任一 zone 觸頂或總數 > 閾值×K → 自動落地（含正常張基線）。
4. 週期 stats line（每 N 張：fps/gpu_ms p50/p95/queue 峰值）→ 效能退化有基線、hang 有「最後活著時間」上界。

**T2（中等投入）**：5. stall watchdog（心跳逾時落 incident）；6. signal handler（SIGSEGV/SIGBUS）；7. RDMA 路徑補 incident（payload>slot、seq 跳躍、credit 低水位——或刪掉不實註解）；8. 修 record_incident 無鎖讀（P1-8）。

**T3（跨模組，補最大結構洞）**：9. Grab 最小行車紀錄（~100 行 jsonl：cam open/close、drop 跳變、send fail、GRAB_START/STOP）；10. Control 命令 trace＋三端以 panel_id+frame_seq 貫通（半夜 debug 從「三台機器考古 console」變「三份 jsonl join」）。

**附帶**：docs 只列 5 種 incident kind、實碼 10 種；incident-viewer.html:111 stale「L1」標註。
